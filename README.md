# hydra

> A self-improving skill library for AI coding agents. Every unsolved problem becomes a
> new head (skill). Cut one off, two grow back — the "do it inline" branch is the
> cauterizing iron that stops it sprawling into useless heads.

**hydra** is a single self-contained Go CLI that curates a library of reusable *skills*
for Claude Code and other agent runtimes. Instead of solving the same recurring problem
from scratch every session, an agent working in a hydra-enabled project first runs a
**curator loop**: it scans the existing skills, then decides whether to use one, sharpen a
weak one, build a new one, or just handle the task inline. The deliverable of most sessions
is a skill — a capability that pays off the next time the problem shows up.

## How it works

A `UserPromptSubmit` hook injects a nudge on every prompt, reminding the agent to run the
curator loop before doing real work:

```
scan → decide → (build | update | use | inline) → sync → log → do the task
```

- **scan** — check the skills already in the library for one that covers the task.
- **decide:**
  - covered well → **use** it
  - no match, *plausibly recurring* and *non-trivial* → **build** a new skill
  - matches but *demonstrably weak* (missing a step, stale, wrong) → **update** it
  - trivial / one-off / unlikely to recur → **do it inline**, create nothing
- **sync** — `hydra sync` exposes `.hydra/skills/*` to every runtime via symlinks.
- **log** — append one line to `.hydra/curator.log` recording what changed and why.

The guard against sprawl: when you're unsure whether something is reusable, do it inline.

## Layout

Running `hydra init` in a project scaffolds a `.hydra/` directory (or `~/.hydra/` with
`--global`):

| Path | Purpose |
| --- | --- |
| `.hydra/skills/<name>/SKILL.md` | **Source of truth** for each skill. Edit here. Supporting scripts/references live in the same dir. |
| `.claude/skills/`, `.agents/skills/` | Symlinks generated by `hydra sync`. Never edit directly. |
| `.hydra/curator.log` | One line per skill created/updated/renamed. |
| `CLAUDE.md` / `AGENTS.md` | Agent-facing operating instructions (mirror copies). |

The CLI itself is a single Go binary; the source of truth is the `*.go` files plus the
`assets/` directory (embedded via `//go:embed`). There is no asset build step.

## The seeded skill

`hydra init` seeds one skill into every project — **`skill-curator`** — which drives the
curator loop above (scan → decide → build/update/use/inline). From there you grow your own
library with `hydra new <name>`; each project's skills live under its own `.hydra/skills/`.

## Install the `hydra` CLI

`hydra` is a single self-contained Go binary.

```bash
go install github.com/webteractive/hydra@latest   # from source (needs Go 1.26+)
# or download a prebuilt binary from Releases onto your PATH
```

### Add the curator to a project

```bash
cd ~/path/to/project
hydra init       # scaffolds ./.hydra/, wires the hook, seeds skill-curator
hydra doctor     # confirm wiring
```

### Or globally (every project)

```bash
hydra init --global   # library in ~/.hydra/, wired into ~/.claude + ~/.agents
```

Open the project in Claude Code (or any compatible agent). The curator hook fires
automatically; skills become available once synced. A freshly created skill won't appear
in the injected metadata list until the next session (`/reload-plugins` refreshes it), but
you can invoke it by name in the current session once synced.

### Commands

| Command | What it does |
|---|---|
| `hydra init [--global]` | Scaffold the curator (idempotent). |
| `hydra sync [--global]` | Rebuild skill symlinks from `.hydra/skills/`. |
| `hydra new <name> [--global]` | Scaffold a new skill, then sync. |
| `hydra log <CREATE\|UPDATE\|RENAME> <skill> <reason>` | Append to `.hydra/curator.log`. |
| `hydra doctor [--global]` | Verify install health. |

## Adding a skill

1. Author it via the `skill-creator` skill (don't hand-roll the structure), or scaffold
   one with `hydra new <name>`.
2. Each `SKILL.md` starts with YAML frontmatter — `name` (kebab-case, matching the dir) and
   a `description` that says *when to use it*. That description line is what future sessions
   match against, so make it specific.
3. Keep skills focused, imperative, and lean. One skill = one coherent capability.
4. Prefer documenting *verified* command surfaces (note the tool version you checked) over
   guessed flags.
5. Run `hydra sync`, then log the change with `hydra log`.

## Developing hydra

Source of truth: `*.go` + `assets/` (embedded via `//go:embed`). No build step for assets.

```bash
go test ./...    # full suite
go vet ./...
go build -o hydra .
```

## Conventions

- **`CLAUDE.md` and `AGENTS.md` are mirrors** — they share one body (only the title/intro
  differs). Any edit to one must be replicated to the other in the same change.
- **Never auto-commit or push.** Write and sync freely; the git diff is the review gate.
- Scope is project-local by default; pass `--global` to any command to manage the library
  in `~/.hydra/`, wired into `~/.claude` and `~/.agents`.
