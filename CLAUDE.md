This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

# skiller — self-improving skill builder/curator

This project curates a library of skills. Before any real task, the `skill-curator`
skill decides whether to build a new skill, update a weak one, or just use what exists.
The deliverable of most sessions is a *skill*, not a one-off answer — when you solve a
recurring, non-trivial problem here, capture it as a skill so it pays off next time.

## Layout
- `skills/<name>/SKILL.md` — **source of truth** for every curated skill. Edit here.
  Any supporting files (scripts, references) live in the same `skills/<name>/` dir.
- `.claude/skills/`, `.agents/skills/` — symlinks generated by `bin/sync.sh`. Never edit
  these directly; edit the source in `skills/` and re-run sync.
- `hooks/curator-reminder.sh` — `UserPromptSubmit` hook (wired in `.claude/settings.json`)
  that injects the curator nudge on every prompt.
- `bin/sync.sh` — exposes `skills/*` to both runtimes (idempotent, prunes dangling links).
- `log/curator.log` — append one line per skill created/updated.
- `docs/plans/` — design docs (e.g. the curator design + rationale).

There is no build/lint/test tooling and no package manager — this is a skills + bash repo.
"Testing" `sync.sh` means running it twice (idempotent) and confirming links resolve in
both target dirs.

## The curator loop (run before real work)
`scan → decide → (build | update | use | inline) → sync → log → do the task`

1. **Scan** the skills in your injected metadata for one that covers the task.
2. **Decide:**
   - Covered well → **use** it.
   - No match, AND *plausibly recurring* AND *non-trivial* → **build** a new skill.
   - Matches but *demonstrably weak* (missing a step, stale, wrong) → **update** it.
   - Trivial / one-off / unlikely to recur → **do it inline**, create nothing.
   Guard against sprawl: when unsure whether it's reusable, do it inline.
3. **Author** via the `skill-creator` skill (new) or `superpowers:writing-skills` (edit) —
   don't hand-roll skill structure.
4. **Sync** immediately: `bin/sync.sh`.
5. **Log** to `log/curator.log` (format: `DATE  ACTION  skill-name  — reason`; ACTION is
   `CREATE` / `UPDATE` / `RENAME`).

## Authoring conventions
- Every `SKILL.md` starts with YAML frontmatter: `name` (kebab-case, matches the dir) and a
  `description` that says *when to use it* — that line is what future sessions match against.
- Keep skills focused, imperative, lean. One skill = one coherent capability.
- Prefer documenting *verified* command surfaces (note the tool version you checked against)
  over guessed flags.

## Rules
- **`CLAUDE.md` and `AGENTS.md` are mirrors — keep them in sync.** They share one body
  (only the top title/intro line differs). Any edit to one MUST be replicated to the other
  in the same change. Never let them drift.
- **Never auto-commit or push.** Write and sync freely; the git diff is the review gate.
  Ask before `git commit`.
- After creating or editing a skill, run `bin/sync.sh`.
- A freshly created skill won't appear in the injected metadata list until the next session
  (`/reload-plugins` refreshes it), but you can invoke it by name in the current session
  once synced.
- Scope is project-local for now. Promoting to global = point `sync.sh` at `~/.claude/skills`
  + `~/.agents/skills` and copy the hook into `~/.claude/settings.json`.
